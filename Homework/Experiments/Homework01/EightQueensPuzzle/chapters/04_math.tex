\section{运行时间比较}

以 M2（纯逻辑 \texttt{permuteq}）作为参考基线，定义加速比：
\begin{equation}
  \text{Speedup}(M_i)=\frac{T_{M2}}{T_{M_i}}.
\end{equation}

代入实测均值可得：
\begin{itemize}
  \item M1 相对 M2 加速约 $9.37\times$；
  \item M3 相对 M2 加速约 $5.78\times$；
  \item M4 与 M5 相对 M2 分别加速约 $2749.82\times$ 和 $2797.83\times$；
  \item M6 相对 M2 加速约 $26840.96\times$。
\end{itemize}

如果只在搜索类实现里比较，M6 相对 DFS、BFS 仍有约 $9.76\times$ 与 $9.59\times$ 的优势。这说明在搜索树规模接近时，位运算主要赢在“每个节点处理得更快”。

\section{搜索规模与剪枝效果}

状态规模反映了“有多少分支真的被走到”。M1 与 M2 都遍历 40320 个全排列候选；M3 降至 15720，说明增量约束确实提前剪掉了大量无效路径；M4/M5/M6 都是 2057，说明三者的搜索树规模同阶。

一个值得强调的现象是：M4/M5/M6 的节点数一样，但耗时差别明显。原因不在搜索规模，而在实现细节。M6 的冲突检测和状态推进都在位级完成，减少了 Python 层循环与对象操作带来的常数开销。

需要注意，复杂度上界只回答“增长会不会很快”，并不直接给出“实际跑多快”。本实验中，约束触发顺序、状态表示方式和解释器常数共同决定了最终耗时。

\section{本章小结}

本章数据支持两个结论：第一，六种实现在正确性上彼此一致；第二，性能差异主要来自两个层面，即剪枝是否足够早、节点处理是否足够轻。前者决定要探索多少状态，后者决定每个状态要花多少时间。
